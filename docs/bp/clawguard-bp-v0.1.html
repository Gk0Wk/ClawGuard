02 / Problem

OpenClaw 越火，安全控制问题越不能再留给用户自己扛

OpenClaw 正快速走向普通用户与小团队，但用户面对的是新的风险结构：AI 可能被诱导、可能越权、可能泄密，而用户往往并不知道它下一步要做什么。

  • 一句群消息，可能诱导 AI 执行危险动作
  • 用户看不清、拦不住、事后追不回
  • 关键动作发生前，缺少明确的刹车机制
  • 风险记录、责任链与解释权没有产品化
核心痛点 它太听话、权限太大，而用户没有控制面。
03 / Insight

普通用户真正需要的，不是安全术语，而是安全感、控制感和解释权

这类需求很像 PC 时代安全卫士的起点：大部分用户不懂底层安全原理，但他们需要有人告诉自己“有没有风险、风险在哪、现在该怎么办”。

先看见

先告诉用户现在有没有风险、风险等级是多少。

再理解

把复杂风险翻译成普通人能理解的语言。

再决定

关键动作发生前，把最终确认权交还给人。

时代类比 从 PC 安全卫士，到 AI Agent 安全卫士。
04 / Solution

ClawGuard = OpenClaw 的安全控制层

ClawGuard 面向 OpenClaw 生态，提供一套“先解释、再审批、能阻断、可审计”的安全控制能力。

危险动作审批

删除文件、外发消息、执行命令等关键动作先过人。

风险体检与安全评分

让用户知道当前实例的总体安全状态。

skills 验毒

对扩展与能力包做安装前风险判断。

基础泄密阻断

阻止敏感信息被外发，并保留风险解释。

产品价值 安全性 + 安全感 + 控制权
05 / Demo

一句群消息，差点让 OpenClaw 发红包；ClawGuard 把最后一票还给了人

这是项目的 north star demo，也是最容易被普通用户、媒体与合作方一眼理解的故事。

Step 1
群聊中出现一条诱导消息，试图让 OpenClaw 执行资金类动作。
Step 2
OpenClaw 进入高风险动作准备态。
Step 3
ClawGuard 在执行前识别风险、解释原因、进入审批流。
Step 4
用户决定拒绝，危险动作不执行，审计页保留完整记录。
我们要证明的不是 AI 永不犯错 而是关键动作不能黑盒发生。

差异化 我们抢的不是单点检测能力，而是安全解释权与控制权入口。
